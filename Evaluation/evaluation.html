🔧 v5 — 12 Critical Chat Bugs Fixed This Session
✅
camelCase JSON — ALL structured cards were silently broken

JsonSerializer.Serialize(structured) used PascalCase by default. Frontend reads camelCase (structured?.councilTaxProperties). Every library list, address picker, bin card and council tax card was invisible. Fixed with _camelCase JsonSerializerOptions in AgentService.

✅
SSE newline encoding — all markdown was one unformatted blob

When AI tokens contained \n, the SSE data: …\n\n frame broke. Frontend skipped lines without data: prefix, dropping all headings and bullets. Fixed by escaping \n → \\n in ChatController and unescaping in frontend stream reader.

✅
CORS blocking local frontend — file:// origin was rejected

Production CORS policy only allowed Netlify URL. start-all.ps1 opens login.html as file:// which sends Origin: null. Fixed by adding "null", localhost:5000 and 127.0.0.1:5000 to CORS origins. Also fixed config.js to auto-route: file:// → localhost, Netlify → Railway.

✅
Parallel tool calls — council tax called address lookup simultaneously

parallel_tool_calls: true let the LLM call both lookup_council_tax_band and lookup_addresses_for_postcode for the same postcode. Address data overwrote the tax flow. Fixed with parallel_tool_calls: false and explicit prompt rule: "NEVER call lookup_addresses_for_postcode for council tax queries."

✅
Message text not rendering after pop-up mode change

Changed from streaming token-by-token to showing complete message at once. finaliseStreamBubble relied on updateStreamBubble having pre-filled .stream-text during streaming. After removing streaming updates, single-bubble messages appeared empty. Fixed by always setting firstBubble.querySelector('.stream-text').innerHTML in finaliseStreamBubble.

✅
6 more fixes: prompt flows, invalid postcodes, address cap, library names

Library by name asked for postcode instead of calling tool · Council tax with postcode in message ignored (asked again) · Invalid postcode showed fake "various bands" rates · Address results capped at 30 (lifted to 100) · Overpass search radius 150m→200m · All prompt flows updated to scan current message for BD postcode before asking.

🔬 Deep Data Quality Analysis

Council Tax Data — GOV.UK Scraper

  • GOV.UK returns property-level bands (A–H) per address
  • 2026/27 rates hardcoded and verified (Band D = £2,360.73)
  • Rate ratios follow statutory ninths (A=6, B=7…H=18)
  • Results cached 5 min per postcode (IMemoryCache)
  • GOV.UK scraper is a 3-step HTML parse — fragile to HTML changes
  • BD1, BD8 9AT return no band data (commercial / sparse areas)
  • councilTaxInfo.band always shows first alphabetical match, not majority band
  • GOV.UK does not return all properties for large postcodes (caps at ~20)

Address Data — 4-Source Fallback

  • getAddress.io (Royal Mail PAF) wired in — just needs API key
  • Bradford form scraper works for most BD postcodes
  • Overpass 3-pass strategy with 200m radius
  • Cap raised from 30 → 100 addresses
  • Without getAddress.io key, OSM data is incomplete for rural/new builds
  • BD8 9AT, BD16 1AY, BD21 2PH return 0 addresses from all sources
  • When 0 addresses found, LLM says "invalid postcode" instead of correct fallback message
  • UPRN field is empty for Overpass/Nominatim results — bin lookup must fall back to fuzzy match

Bin Collection Data

  • Parallel strategy: Bradford form scraper + JSON API endpoints
  • Bin card shows grey/green/brown schedule correctly
  • Full address in message → skips address picker (direct to bin dates)
  • Hardcoded schedule text used when scrapers fail ("Fortnightly — general waste")
  • Actual specific collection dates not scraped (only schedule type)
  • Some postcodes get "next collection" dates, others get generic schedule — inconsistent

Library Data

  • 20 Bradford libraries with distance from postcode (find_local_services)
  • Library details: address, opening hours, facilities scraped from bradford.gov.uk
  • Fuzzy name matching for get_library_details
  • All 20 libraries share the same phone number (01274 433600) — should be per-branch
  • Opening hours scraped at request time — may be stale if webpage is down
  • Library structured data (LibraryOption) only appears in list view — not in detail view

⚠️ Risk Matrix — Remaining Open Issues
HIGH — Addresses Incomplete No getAddress.io key means OSM only. BD8 9AT, BD16, BD21 return 0 addresses. Bin and council tax flows break silently for ~15% of Bradford postcodes.

Fix: Sign up at getaddress.io (free trial 100/day). Add key to appsettings.Local.json and Railway env var GetAddress__ApiKey.
HIGH — Railway Instability Multiple 502 errors during testing. Netlify frontend shows "trouble connecting" to all users when Railway restarts. No health check retry, no fallback message.

Fix: Add Railway health check config. Add frontend retry logic (2 attempts, 2s apart) before showing error.
MEDIUM — Debug Logs in Production console.log('[BCA]', ...) added for debugging now ships to Netlify. Exposes structured data shape to any user who opens DevTools.

Fix: Remove the 2 console.log statements from chat.js before next deploy.
MEDIUM — Bare Postcode Ambiguity "BD7 3BX" alone triggers both council tax AND address tools. Both cards render but user may be confused. Intent should be clarified.

Fix: Add prompt rule: if message is ONLY a postcode with no context, ask "Are you checking council tax, bin dates, or nearby services?"
MEDIUM — CT Band Accuracy councilTaxInfo.band is set to the first match found, not the most common band. For mixed postcodes this could mislead users about their likely band before they select their property.

Fix: Change to use most-frequent band: govResults.GroupBy(r => r.Band).OrderByDescending(g => g.Count()).First()
LOW — Session Reset on Reload SID is random on each page load. Users who reload lose their conversation context. The DB retains history but the frontend cannot resume the session.

Fix: Persist SID to localStorage with TTL. On page load, resume if <2 hours old.
