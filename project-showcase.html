Project Overview

What was built, why it matters, and how it works at a high level.

🎯

The Problem

Bradford Council's website has hundreds of pages across dozens of services. Residents waste time navigating menus, calling helplines, or failing to find information at all — especially for common queries like bin dates, council tax, and school admissions.

🤖

The Solution

A single conversational interface that understands natural language and autonomously calls the right council data source. Ask anything — the AI agent decides which tool to invoke, scrapes or queries the relevant service, and returns a structured response with rich UI cards.

📈

The Impact

Zero phone calls needed for bin dates, council tax lookups, school finders or library queries. The assistant answers in under 5 seconds what previously required navigating 3–5 council web pages or waiting on hold for 10+ minutes.

Frontend Layer

Pure HTML/CSS/JavaScript — no framework, no build step. config.js auto-detects the environment: file:// routes to localhost:5000, any https:// host routes to Railway. SSE tokens stream directly into the DOM.

HTML5 Vanilla JS SSE Vercel CDN

Backend Layer

ASP.NET Core 8 Web API with two endpoints: a synchronous POST /api/chat and a streaming GET /api/chat/stream. Rate limiting (10 req/min) and CORS protect the API.

ASP.NET Core 8 Docker Railway

AI / Agent Layer

GPT-4o-mini with function calling. The agent loops up to 3 rounds — LLM decides which tool to call, the tool executes and returns structured data, the LLM synthesises the final answer.

OpenAI GPT-4o-mini Tool Calling SSE Streaming

Agent Loop & Tool Execution Flow

How the AI agent decides what to do and how tools are selected and executed

1

User sends message

e.g. "When is my bin collected? BD5 8LT" — the message is saved to SQLite conversation history alongside the last 20 turns for context.

2

Context preparation

AgentService checks if the query is tool-based (postcode, bin, school, tax keywords). If yes — RAG is skipped to save 200–800ms. If no — top-3 Qdrant chunks are fetched and injected into the system prompt.

3

LLM call with tool definitions

All 12 tool definitions are sent to GPT-4o-mini alongside the conversation history. The model decides whether to respond directly or call a tool — it can only call one tool per round (parallel_tool_calls: false).

4

Tool execution

CouncilToolService receives the tool name and JSON arguments. It scrapes the appropriate council data source — e.g. for bin dates: GET Bradford eForms, extract ViewState, POST with postcode, parse ASP.NET GridView HTML response.

5

Structured data markers

Tool results embed markers like [[BIN_DATE_CARD]]...[[/BIN_DATE_CARD]]. AgentService.ExtractStructuredData parses these from conversation history and attaches typed objects (BinDateCard, SchoolCard, etc.) to the ChatResponse.

6

Stream to frontend

The final LLM answer streams token-by-token via SSE. A final [STRUCTURED] SSE event carries the typed card data as camelCase JSON. The frontend renders rich cards (bin schedule, school list, council tax table) alongside the text reply.

Tool Selection System Prompt (excerpt)

## Bin collection — strict flow
1. Scan the CURRENT message for a Bradford postcode (BD…). If present → go to step 2.

Data Scraping & Integration Layer

How the system extracts real data from Bradford Council's legacy web systems

🚪 Bin Dates — Bradford eForms

Bradford uses a Verj.io eForms system. Standard HTTP POST failed due to ASP.NET ViewState validation and JavaScript-set navigation fields.

Solution: GET the form to extract __VIEWSTATE, __EVENTVALIDATION, and navigation fields (PAGE:N.h, APAGE:N.h). POST with the exact control names the ASP.NET WebForms app expects. Parse the GridView HTML response using HtmlAgilityPack.

ASP.NET ViewState HtmlAgilityPack Cookie sessions

🏫 School Finder — Bradford BSO

Bradford's school finder is embedded as an iframe from bso.bradford.gov.uk — URL parameters are ignored, results are server-rendered.

Solution: Scrape bso.bradford.gov.uk/council/schools/schoolfinder.aspx directly. POST with ctl00$ContentPlaceHolder1$tbPostcodeArea. Distances from BSO are unreliable — use postcodes.io bulk API to calculate haversine distances from school postcodes. Sort nearest-first.

ASP.NET WebForms postcodes.io bulk Haversine formula

💷 Council Tax — GOV.UK VOA

Uses the Valuation Office Agency's public search. Extracts band (A–H) and calculates Bradford's annual and monthly charge from known rate tables.

Falls back gracefully — if no band found, does not emit a council tax card to avoid showing incorrect data.

GOV.UK VOA Rate lookup table

Distance Calculation (Schools Example)

// 1. Get user lat/lon from postcodes.io
var (userLat, userLon) = await GetLatLonAsync(postcode, ct);

// 2. Extract school postcodes from BSO results
var schoolPostcodes = schools.Select(s => ExtractPostcode(s.Address))
                             .Where(p => p != null).Distinct().ToList();

// 3. Bulk lookup all postcodes in ONE API call (up to 100 per request)
var pcMap = await BulkLookupPostcodesAsync(schoolPostcodes, ct);

// 4. Calculate haversine distance for each school
foreach (var school in schools) {
    var pc = ExtractPostcode(school.Address);
    if (pc != null && pcMap.TryGetValue(pc, out var ll)) {
        var mi = HaversineDistanceMi(userLat, userLon, ll.lat, ll.lon);
        school.Distance = $"{mi:F1} mi";
    }
}

// 5. Sort by distance
schools = schools.OrderBy(s => ParseDistanceValue(s.Distance)).ToList();

Frontend Architecture & Rich UI Cards

How the chat interface renders structured responses and manages state

SSE Streaming Pipeline

The frontend opens a GET SSE connection. Tokens arrive line-by-line and are appended to the chat bubble in real time. When the stream ends with [STRUCTURED], the JSON payload is parsed and the appropriate card is injected below the text bubble.

// Parse SSE lines
if (data.startsWith('[STRUCTURED]')) {
  structured = JSON.parse(data.slice(12));
} else {
  fullText += data.replace(/\\n/g, '\n');
}
// After stream ends
finaliseStreamBubble(div, fullText,
  structured?.addresses,
  structured?.binDates,
  structured?.schools,
  structured?.schoolDetails);

Presentation Pitch

Use these sections to present the project. Each part is timed — total around 5 minutes.

⏱ 45 seconds — Opening Hook

Part 1 — The Problem

Imagine you're a Bradford resident. It's Tuesday evening. You need to know if your bin goes out tomorrow. You go to bradford.gov.uk, navigate three menus, fill in a postcode form, click through an address picker — and if you're lucky, you find the answer five minutes later. Now imagine doing that for council tax, school admissions, library opening hours, and planning applications.

Bradford Council's website serves over half a million residents. It has hundreds of pages, multiple legacy systems, and no single place to get a fast answer. The result? Frustrated residents, overloaded phone lines, and council staff answering the same basic questions every day.

⏱ 90 seconds — The Solution

Part 2 — What I Built

I built the Bradford Council AI Assistant — a full-stack conversational AI that gives residents instant, accurate answers about any council service from a single chat window.

The system uses OpenAI GPT-4o-mini with an agentic tool-calling loop. When a user asks "when is my bin collected?", the AI doesn't just search the web — it autonomously identifies the relevant council data source, scrapes the Bradford eForms system in real time, and returns the exact upcoming dates for that specific address.

The architecture is a clean separation of concerns: a Vanilla JS frontend deployed on Vercel, an ASP.NET Core 8 API on Railway, and 12 purpose-built tools that scrape or query Bradford's actual council systems. Responses stream token-by-token using Server-Sent Events, and rich UI cards — bin schedules, council tax tables, school lists with Ofsted ratings — appear automatically alongside the text.

The most challenging part was the web scraping layer. Bradford's systems use ASP.NET WebForms with ViewState validation, Verj.io eForms with JavaScript-driven navigation, and iframe-embedded applications that don't respond to URL parameters. I reverse-engineered each form, identified the exact hidden field names and navigation tokens each system expects, and built scrapers that work reliably in production.

⏱ 60 seconds — Technical Achievement

Part 3 — What Makes It Technically Strong

This isn't a chatbot that wraps a search engine. It's an autonomous agent with real data access. The LLM receives 12 tool definitions and decides — based on the user's message — which to call, in what order, with what arguments. The agent loops up to three rounds: call a tool, get a result, decide if another call is needed.

Key technical decisions: parallel_tool_calls is disabled to prevent race conditions between address lookup and bin date retrieval. RAG is bypassed for tool-based queries to cut 400ms of latency. Structured data markers in tool results are parsed server-side and delivered as typed objects in the same SSE stream, so the frontend renders cards without a second API call.

The system is live in production at bradford-council-ai.vercel.app — not a demo, not a mockup. Real Bradford Council data, real addresses, real school distances calculated with the Haversine formula from postcodes.io bulk lookups.

⏱ 45 seconds — Closing

Part 4 — Impact & What's Next

The assistant answers in under 5 seconds what previously required navigating 3–5 council web pages or waiting on hold. It's accessible 24/7, works from any browser without installation, and handles follow-up questions because it maintains conversation history in SQLite.

What I'd do next: integrate with Bradford's live Alloy waste management API when available, add Welsh language support for Bradford's diverse communities, build a council staff-facing version that can process benefit applications through the same conversational interface, and add voice input for residents with accessibility needs.

This project demonstrates that AI can make public services genuinely accessible — not by replacing council workers, but by giving residents a direct line to the information they need, when they need it.
